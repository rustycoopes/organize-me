## Feature Workflow

Requirements-gathering and planning are agentic, skill-driven steps — not something to skip
straight past into code for anything non-trivial. See
[docs/feature-workflow.md](docs/feature-workflow.md) for the full walkthrough of each stage,
including which agents each one uses.

- **Minor changes** (docs, small fixes, one-obvious-change work): skip straight to a GitHub issue
  (or straight to a branch) — no feature directory needed.
- **Major changes** (spans more than one deployable slice, or needs a real design/architecture
  decision — new schema, new external integration, a new seam): work through the full pipeline
  below before implementation starts.

```
/grilling → /to-prd → /to-design → /to-wbs → /to-issues → /to-implementation
```

1. **`/grilling`** — interview to pin down requirements. This is the only interview step; every
   skill after it assumes the groundwork is already done.
2. **`/to-prd`** — synthesizes the conversation into `docs/features/<feature-slug>/PRD.md`.
   No interview, no issue-tracker interaction.
3. **`/to-design`** — turns the PRD into `docs/features/<feature-slug>/TDD.md` using parallel
   design sub-agents, writing an ADR (`docs/adr/<feature-slug>-<decision-slug>.md`) for every
   decision with a real trade-off.
4. **`/to-wbs`** — breaks the PRD + TDD into vertical (tracer-bullet) slices, quizzes on
   granularity/dependencies, and writes `docs/features/<feature-slug>/WBS/slice-<n>-<name>.md`
   per slice.
5. **`/to-issues`** — takes one approved WBS slice, quizzes on whether it has enough detail for
   `/to-implementation` to start cold, and publishes a GitHub issue.
6. **`/to-implementation`** — implements the issue, appends a `## Delivered` section to the WBS
   slice, and adds the changelog index line.

`/next-issue` picks the next issue to work on across whatever tracks currently have ready work.

### Feature directories

A feature lives in whichever repo it actually concerns: a single-app feature lives in that app's
own repo; a cross-app/platform feature lives here, in `organize-me`.

```
docs/features/<feature-slug>/
  PRD.md
  TDD.md
  WBS/
    slice-1-<name>.md   ← What to build / Design notes / Blocked by / Acceptance criteria /
    slice-2-<name>.md     Testing / Delivered (appended by /to-implementation once merged)
```

`docs/features/platform-restructure/` and `docs/features/original-organize-me/` are the two
pre-existing bodies of work this convention formalizes — moved under `docs/features/` to match,
kept in their original (not the `PRD.md`/`TDD.md`/`WBS/slice-<n>-<name>.md` naming) shape rather
than rewritten.

### ADRs

Architecture decisions live centrally per-repo at `docs/adr/<feature-slug>-<decision-slug>.md` — a
sibling of `docs/features/`, not nested inside a feature directory.

### Labels

Each issue generated by `/to-issues` carries three labels: `<feature-slug>`, `<slice-id>`
(e.g. `slice-2`), and `enhancement` — kept as two separate labels, not combined.

## Git Workflow & Security Rules

- **Strict Environment Security**:
  - Never stage or commit `.env` or `.env.local` files. Make sure they are listed in `.gitignore`.
  - Never commit files containing active API keys or credentials.
- **Minor changes** (docs, small fixes): commit directly to the current branch with a meaningful
  message.
- **Major changes** (new features, structural refactors, significant behaviour changes): create a
  new branch before starting work. Name the branch descriptively (e.g.
  `feature/prompt-versioning`, `refactor/file-storage`, `fix/gemini-error-handling`). Commit
  locally with a meaningful message once the changes are finalised. Do not push or merge without
  being asked.

## GitHub Issues

- When creating GitHub issues, always use the **OrganizeMe** project within GitHub (the GitHub
  Projects board linked to this repository).
- Assign issues to the OrganizeMe project when creating them via `gh issue create`.
